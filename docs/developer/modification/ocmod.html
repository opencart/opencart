OCMOD — OpenCart 4.1 Modification System

OCMOD lets you modify OpenCart's PHP, Twig, language and config files without touching the originals. Instead of editing catalog/controller/... directly, you ship an XML file with <search> / <add> rules. OpenCart applies the rules to a copy of the original file in memory and writes the patched version under extension/ocmod/.

When you build a new module and want to bundle OCMOD modifications with it, don't invent a new path. Drop the .ocmod.xml files into an ocmod/ folder inside your .ocmod.zip package, exactly like the canonical OpenCart examples (see oc_ocmod_example.ocmod.zip).

.ocmod.zip package layout

Minimum:

``` my_module.ocmod.zip ├── install.json ← metadata ├── admin/ ← unpacked into extension//admin/ │ └── ... ├── catalog/ ← unpacked into extension//catalog/ │ └── ... ├── system/ ← unpacked into extension//system/ │ └── ... └── ocmod/ ← NOT unpacked to disk ├── modification1.ocmod.xml ← read by installer.xml() and └── modification2.ocmod.xml ← inserted into oc_modification ```

install.json:

```json { "name": "My module", "description": "What it does", "version": "1.0.0", "author": "You", "link": "https://example.com", "code": "my_module" } ```

Important: OpenCart deliberately skips the ocmod/ directory when unpacking files into extension/<code>/ (installer.install(), lines 431-433). The XML lives only in the oc_modification table. This is by design — it lets the admin UI enable / disable each XML individually.

Canonical examples ship under upload/system/storage/marketplace/:

  • oc_ocmod_example.ocmod.zip — pure OCMOD (only ocmod/ + install.json)
  • oc_payment_example.ocmod.zip — full extension with admin/catalog/system, no ocmod
  • oc_theme_example.ocmod.zip, oc_language_example.ocmod.zip — other variants

XML format

```xml Human readable name What this modification does my_unique_code 1.0.0 Your name https://example.com/your-mod load->language('common/header');]]> ```

<file path="...">

AttributeDescription
path Path relative to the OpenCart root, forward slashes. Multiple paths can be OR'd with |. Glob is supported.

Allowed prefixes: admin/, catalog/, system/, extension/.

<operation error="...">

errorBehaviour when <search> finds no match
skipContinue with the next operation (recommended for optional patches)
break(default) Stop all operations for the current file
abortRoll back the entire modification — all files revert to their original state

<search>

AttributeDefaultDescription
regexfalseIf true, the body is a PCRE pattern (/foo/i)
trimtrueTrim whitespace from the body before matching
index(none)Comma-separated zero-based occurrence indices: 0,2 = 1st and 3rd hits
limit-1(regex mode) maximum number of replacements

<add>

AttributeDefaultDescription
positionreplacereplace, before, after
offset0(plain mode) extra lines to consume / shift relative to the match
trimfalseTrim whitespace from the body before inserting

<ignoreif> (optional)

If the file already contains the given snippet (or matches the regex), the operation is skipped. Useful for repeated refreshes or when another modification has already patched the same file.

⚠️ When patching a PHP class — also rewrite the namespace

OpenCart's event handlers look the patched class up in the OCMOD namespace, not the original. So if the original is Opencart\Admin\Controller\Catalog\Attribute, your modification has to rewrite the patched file's namespace to Opencart\Admin\Controller\Extension\Ocmod\Catalog\Attribute. Then the autoloader finds the patched file when the event handler asks class_exists(), and the route gets rewritten to extension/ocmod/<route>.

Three valid ways to ship an OCMOD modification

1. As part of your own .ocmod.zip package (recommended for new modules)

This is the OpenCart 4.1 idiom. When you build a new extension and want to bundle OCMOD modifications with it, drop them into the ocmod/ folder inside the .ocmod.zip:

``` my_module.ocmod.zip ├── install.json ├── admin/ ← your module's regular files ├── catalog/ ├── system/ └── ocmod/ ├── patch_header.ocmod.xml └── patch_checkout.ocmod.xml ```

The Admin > Installer workflow handles the rest:

  • The module's files land in extension/<code>/
  • The XMLs are inserted into oc_modification (linked to your extension_install_id, so they're cleaned up when the module is uninstalled)
  • Click Modifications > Refresh and the patched files are generated

2. Standalone .ocmod.zip (modification only, no extension code)

``` my_patch.ocmod.zip ├── install.json └── ocmod/ └── patch.ocmod.xml ```

Upload it through Installer like any other package. Inserted into oc_modification, linked to its extension_install_id, but no files on disk.

3. Developer drop-in (for fast iteration)

Put *.ocmod.xml straight into upload/system/:

``` upload/system/dev_test.ocmod.xml ```

Refresh picks it up automatically — no installer, no DB row. Delete the file and refresh again to roll back.

Known limitations in OC 4.1

  1. Catalog extension OCMOD overrides cannot be loaded transparently. catalog/controller/startup/extension.php only registers the global extension/ocmod path, not per-extension extension/ocmod/extension/<code> paths the way the admin counterpart does. Patches under extension/<code>/catalog/... get written to extension/ocmod/extension/<code>/catalog/... by refresh, but the autoloader, language and template engines won't find them at runtime in the storefront. Patch the extension's own files in place, or stage your patch on the admin side.

  2. Helpers (system/helper/...) are not event-driven. Loader::helper() uses include_once directly without firing any event. Helper-file patches are written to extension/ocmod/system/helper/<x>.php by refresh but never loaded. If you must override a helper, ship a replacement function in your own extension and load it earlier than the core helper.

  3. Top-level files cannot be patched. cron.php, index.php, config.php, .htaccess and other repository-root files load before the event system or sit outside it entirely.

Common pitfalls

  • Always run Refresh after changing XML. Without it the patched files don't exist, and the autoloader has nothing to pick up.
  • Keep <search> blocks short and unique — line-by-line matching uses case-insensitive stripos(); long blocks are brittle to whitespace differences.
  • Use error="skip" for optional patches and error="abort" only when a partial apply would leave the file broken.
  • Do not patch files inside extension/ocmod/ — that's the output directory and is wiped on every refresh.
  • Cache invalidation: after refreshing a Twig template patch, clear system/storage/cache/ and the Twig template cache; otherwise stale compiled templates keep pointing at the old contents.
  • OpenCart upgrades: after upgrading core, run Refresh again so your modifications re-apply to the new source.
