Models

A model holds the persistence and domain logic for a feature. Models are loaded on demand by the framework and exposed on the controller as $this->model_<path>.

Models extend \Opencart\System\Engine\Model.

Loading a model

```php $this->load->model('setting/setting'); $this->model_setting_setting->editSetting('config', $data); $value = $this->model_setting_setting->getSetting('config'); ```

The path slashes become underscores in the property name (setting/setting → $this->model_setting_setting). Calling $this->load->model(...) twice with the same route is cheap -- the loaded instance is cached.

The proxy: why method calls go through events

When you load a model, the framework does not give you the real object. It gives you a \Opencart\System\Engine\Proxy with one closure per public method. Each closure dispatches:

  1. model/<route>/<method>/before -- listeners can rewrite the args (or the route)
  2. the actual method on the underlying model
  3. model/<route>/<method>/after -- listeners can rewrite the return value

This is what lets OCMOD redirect setting/setting.editSetting to a patched copy at extension/ocmod/setting/setting. From your code's point of view it is invisible.

What is available on $this

The same registry-backed services as a controller ($this->db, $this->config, $this->cache, $this->load, etc.). A model can load other models, fire events, or query the cache exactly like a controller.

Minimal example

```php db->query("SELECT `message` FROM `" . DB_PREFIX . "demo_hello` WHERE `hello_id` = '" . (int)$id . "'"); return (string)($query->row['message'] ?? ''); } } ```

Notes:

  • Always cast or escape values that go into a SQL string. $this->db->escape($value) for strings, (int) / (float) for numbers.
  • Prefix table names with DB_PREFIX. The constant is set from config.php and is required for multi-install support.
  • Return plain arrays/scalars rather than entity objects -- the rest of OpenCart treats results as arrays.
