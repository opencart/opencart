Events

OpenCart fires events at well-known hook points so extensions can observe or rewrite framework calls without monkey-patching the core. This is the same mechanism OCMOD uses to redirect routes to patched copies under extension/ocmod/.

What the framework fires

system/engine/loader.php emits a before trigger before each load and an after trigger when the load succeeds. The exact trigger string is built from the load type and the route, e.g. controller/checkout/cart/before.

Loader callBefore triggerAfter triggerBy-reference args
$this->load->controller($route)controller/<route>/beforecontroller/<route>/after$route, $args, $output
$this->load->model($route) (lazy proxy)model/<route>/beforemodel/<route>/after$route, $args, $output
$this->load->view($route, $data)view/<route>/beforeview/<route>/after$route, $data, $code, $output
$this->load->language($route, $prefix, $code)language/<route>/beforelanguage/<route>/after$route, $prefix, $code, $output
$this->load->library($route, ...$args)library/<route>/beforelibrary/<route>/after$route, $args, $object
$this->load->config($route)config/<route>/beforeconfig/<route>/after$route, $output

Helpers ($this->load->helper()) do not fire events.

The after trigger only fires if the load succeeded -- a failed factory lookup throws and never reaches the second event.

Registering a listener

There are two ways a listener gets attached to a trigger pattern:

  1. Statically through the application config. system/config/admin.php and system/config/catalog.php declare an action_event array that the framework reads on every request and registers verbatim with the Event registry.
  2. Dynamically from the database. The oc_event table holds rows that admin/controller/startup/event.php and catalog/controller/startup/event.php read at startup; each enabled row becomes one registration. This is how marketplace-installed extensions (and the admin Events screen) hook in.

Both paths converge on the same primitive:

```php $event->register($trigger, new \Opencart\System\Engine\Action($action), $priority); ```

where $trigger is a glob-style pattern like controller/*/before and $action is a route to a method on a controller class such as event/modification.controller.

Config-driven listeners

Excerpt from system/config/admin.php:

```php $_['action_event'] = [ 'controller/*/before' => [ 0 => 'event/modification.controller', 1 => 'event/language.before' ], 'controller/*/after' => [ 0 => 'event/language.after' ], 'model/*/before' => [ 0 => 'event/modification.model' ], 'view/*/before' => [ 0 => 'event/modification.view', 999 => 'event/language' ], 'library/*/before' => [ 0 => 'event/modification.library' ], 'language/*/before' => [ 0 => 'event/modification.language' ], 'language/*/after' => [ 0 => 'startup/language.after' ] ]; ```

Catalog has the same shape with a few extra entries (event/theme, event/translation).

The integer key on the inner array is the priority -- lower runs first.

Database-driven listeners

The oc_event row schema (system/helper/db_schema.php):

ColumnDescription
codeInternal identifier (your choice)
descriptionFree-form text shown in admin
triggerTrigger pattern, e.g. catalog/view/common/header/before or admin/controller/sale/order/before
actionRoute to the handler method, e.g. extension/my_module/event/observer.handler
status1 = active, 0 = disabled
sort_orderPriority (lower runs first)

Both startups split the row's trigger on / and look at the first segment:

  • admin/... -- the segment is dropped, the rest is registered. Only the admin startup picks these rows up.
  • catalog/... -- the segment is dropped, the rest is registered. Only the catalog startup picks these rows up.
  • system/... -- registered as-is on both sides.

So the same oc_event table can hold listeners scoped to either application or both. An extension can $this->model_setting_event->addEvent(...) at install time and deleteEventByCode(...) at uninstall time.

Writing a handler class

A handler is a regular controller. The only requirement is that the method signature matches the trigger family (see the table at the top of this page).

```php Register it once at install:

```php $this->load->model('setting/event'); $this->model_setting_event->addEvent( 'my_module', 'My module observer', 'catalog/controller/*/before', 'extension/my_module/event/observer.before' ); ```

Arguments by reference and rewriting routes

Every argument in the listener payload is passed by reference. A listener can therefore change what the framework will load next. This is exactly what event/modification.php uses to redirect a core route to its OCMOD copy:

```php public function controller(string &$route, array &$args): void { if (str_starts_with($route, 'extension/ocmod/')) { return; } $class = $this->prepareClassName($route, 'Opencart\Admin\Controller\Extension\Ocmod\\'); if (class_exists($class)) { $route = 'extension/ocmod/' . $route; } } ```

Touching $args inside controller/*/before mutates the parameters the controller is about to receive; touching $output inside controller/*/after mutates the response that will be returned to the caller. Use this carefully -- a misbehaving after listener can break unrelated pages.

Wildcards in trigger names

Event::trigger() compiles each registered trigger into a regex with two substitutions:

In the trigger stringBecomes regex
*.*
?. (single character)

So controller/*/before matches controller/anything/at/all/before, and controller/checkout/?/before matches single-segment routes only. The match is anchored at the start of the trigger, not at the end -- be careful with patterns like controller/sale which matches controller/sale/order/before as well.

Priority and ordering

Within a single trigger, listeners run in ascending priority order. The Event class re-sorts the internal array on every register() call, so registration order does not matter -- only the priority value does.

OCMOD listeners deliberately use priority 0 so they run first. event/language uses 998/999 so it can fill in any text the route changed.

Worked examples

Log every admin save

```php namespace Opencart\Admin\Controller\Extension\AuditLog\Event; class Audit extends \Opencart\System\Engine\Controller { public function save(string &$route, array &$args, &$output): void { if (!str_ends_with($route, '.save')) { return; } $this->load->library('log'); $this->log->write(sprintf('%s saved %s', $this->user->getUserName(), $route)); } } ```

Register from your install method:

```php $this->model_setting_event->addEvent( 'audit_log_save', 'Log admin saves', 'admin/controller/*/after', 'extension/audit_log/event/audit.save' ); ```

Inject extra view data on the storefront homepage

```php public function before(string &$route, array &$data, string &$code, string &$output): void { if ($route !== 'common/home') { return; } $data['promo_banner'] = '
Free shipping today!
'; } ```
