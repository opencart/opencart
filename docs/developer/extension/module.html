How to build an OpenCart 4.1 module

An OpenCart extension is a folder under extension/<your_code>/ that mirrors the framework layout and ships an install.json. To distribute it, you zip that folder with the suffix .ocmod.zip and let an admin upload it through Extensions → Installer. This page walks through the layout, the install/uninstall flow, and a working "Hello world" example.

Anatomy of an extension

The folder you ship inside the zip looks like the OpenCart source tree, scoped to one extension code:

``` your_module/ ├── install.json ├── admin/ │ ├── controller/ │ │ └── module/ │ │ └── your_module.php │ ├── model/ │ │ └── module/ │ │ └── your_module.php │ ├── language/ │ │ ├── en-gb/ │ │ │ └── module/ │ │ │ └── your_module.php │ │ └── fr-fr/ │ │ └── module/ │ │ └── your_module.php │ └── view/ │ └── template/ │ └── module/ │ └── your_module.twig ├── catalog/ │ ├── controller/ │ │ └── module/ │ │ └── your_module.php │ ├── model/ │ ├── language/ │ └── view/ └── system/ └── library/ └── your_module/ └── helper.php ```

Only the folders you need have to exist. A pure storefront module has no admin/ tree, a config-only module has no view/, and so on.

The ocmod/ sub-folder (optional) is reserved for OCMOD core patches -- see Including OCMOD patches below and the OCMOD reference.

install.json

A minimal manifest:

```json { "name": "Hello Module", "version": "1.0", "author": "Your Name", "link": "https://example.com" } ```

Namespacing rules

The PSR-4-style autoloader (system/engine/autoloader.php) maps the Opencart\Extension namespace to DIR_EXTENSION (which is system/storage/extension/ after install, upload/extension/ when bundled with the source).

So a controller at extension/your_module/admin/controller/module/your_module.php must declare:

```php namespace Opencart\Admin\Controller\Extension\YourModule\Module; class YourModule extends \Opencart\System\Engine\Controller { // ... } ```

Rules:

  • StudlyCase the extension code in the namespace (your_module → YourModule) and the file/class name (your_module.php → class YourModule).
  • Admin classes live under Opencart\Admin\Controller\Extension\<Code>\...; catalog classes under Opencart\Catalog\Controller\Extension\<Code>\....
  • Library classes live under Opencart\System\Library\<subnamespace>\... (no Extension segment, because libraries are loaded from system/library/).
  • Routes follow the same convention: extension/your_module/module/your_module resolves to YourModule\Module\YourModule.

Install / uninstall flow

When an admin uploads a .ocmod.zip, this is what happens (handled in admin/controller/marketplace/installer.php):

  1. Upload. The zip is moved to system/storage/marketplace/ with the original filename.
  2. Install. The extension is unpacked into system/storage/extension/<code>/. install.json is parsed and a row is added to oc_extension_install.
  3. Run install hook. If your admin controller has a public install() method, it is invoked. This is where you create database tables, register events and crons, set defaults.
  4. OCMOD refresh. Any ocmod/*.ocmod.xml files inside the zip are merged into the current modification cache.

Uninstall reverses steps 3 and 4 (the files stay on disk so you can re-enable later) and removes the oc_extension_install row.

Because steps 3 and 4 only run when an admin clicks Install in the UI, never assume your install hook has been called. The upload step on its own is a no-op as far as your code is concerned.

Surfacing the module in admin

OpenCart 4 lists installed modules under Extensions → Extensions → Modules. Once module_hello shows up in oc_extension, the page automatically lets the admin Install, Edit, or Uninstall it. The Edit button links to extension/hello/module/hello, which is why the file path and namespace must follow the convention exactly.

If you need a top-level menu entry instead, OCMOD-patch admin/controller/common/column_left.php to add a row, or hook controller/common/column_left/before via an event listener and append to $data['menus'].

Including OCMOD patches

If your module needs to alter a core file, ship the rewrite XML inside an ocmod/ folder at the root of the zip:

``` hello/ ├── install.json ├── admin/ └── ocmod/ └── hello.ocmod.xml ```

hello.ocmod.xml uses the same syntax described in modification/ocmod. The installer's "OCMOD refresh" step picks it up automatically -- you do not need to reference it from install.json or call any API.

Packaging the .ocmod.zip

From the parent of your module folder:

``` cd path/to/parent zip -r hello.ocmod.zip hello/ ```

Three things to watch:

  • The zip must contain the module folder, not the contents directly. install.json sits one level below the zip root, inside hello/.
  • The filename must end in .ocmod.zip. The installer rejects plain .zip.
  • Do not zip macOS metadata (.DS_Store, __MACOSX/). Use zip -X or pass --exclude.

Testing without uploading

During development you can skip the zip step and place the folder directly under system/storage/extension/<code>/, then insert a row into oc_extension_install and oc_extension as if the installer had run. The simplest way to do that is to run the installer once with a placeholder zip and then keep editing the unpacked files in place -- the autoloader picks up changes immediately.

For OCMOD patches, after every edit click Extensions → Modifications → Refresh to rebuild the cache. The framework loads patched files only after a refresh.
