Startup pre-actions

Each request goes through a list of pre-actions before the main route runs. They prepare the registry: settings, session, language, customer, currency, tax, marketing, etc. The list lives in system/config/<app>.php as action_pre_action; the implementations are plain controllers under <app>/controller/startup/.

How the framework runs them

From system/framework.php:

```php foreach ($config->get('action_pre_action') as $pre_action) { $pre_action = new \Opencart\System\Engine\Action($pre_action); $result = $pre_action->execute($registry, $args); if ($result instanceof \Opencart\System\Engine\Action) { $action = $result; break; } if ($result instanceof \Exception) { $action = $error; // ... } } ```

Three things to notice:

  1. Pre-actions run in the order they appear in the array. Their order matters -- session must run before anything that touches $this->session, language must run before customer if customer messages need translation, and so on.
  2. If a pre-action returns an Action object, that becomes the request's main action and the loop terminates. startup/login and startup/maintenance use this to short-circuit straight to the login or maintenance page.
  3. If a pre-action throws, the loop swaps the main action for action_error (default: error/not_found).

Declared in system/config/admin.php:

1startup/settingLoads oc_setting rows for store 0 into Config.
2startup/sessionConstructs the configured session driver, restores the cookie, refreshes session lifetime and re-emits the cookie with secure flags.
3startup/languageResolves the active language code, registers language in the registry, loads the default file.
4startup/applicationSets application in the registry to "Admin".
5startup/extensionReads oc_extension and prepends each enabled extension's admin/controller, admin/model, admin/language, system/library, admin/view/template, admin/view/javascript, admin/view/stylesheet paths.
6startup/startupRegisters oc_*-style helper functions and the legacy bootstrap.
7startup/errorWires the error handler so unhandled exceptions become a clean error page rather than a stack trace.
8startup/eventReads oc_event and registers each enabled row with the Event registry. See engine/event.
9startup/sassCompiles the admin theme's .scss entry points to .css on demand.
10startup/loginIf the request is not authenticated and not the login page, returns an Action for common/login, short-circuiting the rest of the chain.
11startup/authorizeConfirms the current user has access to the requested route via the oc_user_group.permission map. Falls through to error otherwise.
12startup/permissionLoads access/modify permission flags for the active user group into the registry so the controller can guard write operations.

Short-circuiting and error handling

Three patterns appear repeatedly:

  • Return an Action to redirect. The pre-action constructs a new \Opencart\System\Engine\Action(...) and returns it. The framework switches the request's main action to that one and stops processing further pre-actions.
  • Throw to error out. Any exception causes the framework to substitute action_error as the main action.
  • Mutate the registry and return null. The default pattern -- the pre-action sets up a service in the registry and lets the chain continue.

Adding your own pre-action

Pre-actions are just controllers, so the file goes wherever a controller would. Two paths:

Inside an extension (recommended)

Drop a controller under extension/<your_module>/admin/controller/startup/check.php and a matching catalog file if you need both sides:

```php namespace Opencart\Admin\Controller\Extension\YourModule\Startup; class Check extends \Opencart\System\Engine\Controller { public function index(): mixed { if (!$this->config->get('your_module_enabled')) { return null; } // ... your work here ... } } ```

Then either modify action_pre_action via OCMOD on system/config/admin.php, or use an oc_event row hooked to controller/*/before to run the same logic at request time. The latter is preferable because it does not require touching the config file.

Patching the core list

If you genuinely need a pre-action to run in a specific position relative to the built-ins, use OCMOD to insert your route into the action_pre_action array in system/config/admin.php or system/config/catalog.php.
